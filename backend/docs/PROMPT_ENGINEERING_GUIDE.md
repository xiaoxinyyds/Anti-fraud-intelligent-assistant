## 提示词设计原理

### 核心元素

```
[系统身份] 
你是一位资深的反诈专家，拥有20年的反诈经验。

[任务定义]
你的职责是分析用户提供的文本内容，判断其中是否存在诈骗风险。

[输出格式]
请以JSON格式返回结果。

[分析标准]
1. 识别常见的诈骗模式
2. 识别紧急警告词
3. 分析文本的紧迫性
4. 评估信息的可信度

[输入数据]
- 用户内容
- 相似的诈骗案例参考
```

### 提示词工程的三个层次

| 层次 | 难度 | 效果 | 适用场景 |
|-----|------|------|--------|
| 基础指令 | 低 | 一般 | 演示/测试 |
| 少样本学习 | 中 | 良好 | 生产环境 |
| 角色扮演+上下文 | 高 | 优秀 | 高精度需求 |

## 4个版本的对比分析

### V1_BASIC（基础版）

```python
【特点】
- 最简洁
- 最直接
- 响应最快

【提示词框架】
1. 身份定位
2. 任务描述
3. 分析标准
4. JSON格式要求

【优点】
✓ 快速响应
✓ 易于理解
✓ 基础判断准确

【缺点】
✗ 准确性不够高（70-80%）
✗ 输出格式不一致
✗ 缺乏示例指导

【何时使用】
- 快速演示
- 简单诈骗检测
- 非关键场景
```

### V2_EXAMPLES（少样本学习 - 推荐）

```python
【特点】
✓ 包含2个详细示例（高风险和低风险）
✓ 示例展示了完整的分析过程和输出格式
✓ 平衡性能和准确性

【提示词框架】
1. 身份定位
2. 任务描述
3. [关键] 示例1：高风险场景
4. [关键] 示例2：低风险场景
5. 待分析内容

【优点】
✓ 准确性高（80-90%）
✓ 输出格式一致
✓ 响应速度快
✓ 生产用最佳选择

【缺点】
✗ 稍长于基础版本
✗ 需要精心设计示例

【何时使用】
→ 生产环境（推荐）
→ 大多数实际应用
→ 需要平衡速度和准确性

【为什么有效】
少样本学习利用了大模型的"学习"能力：
1. 模型通过示例理解任务的"风格"
2. 示例展示了预期的推理过程
3. 示例演示了完整的输出格式
4. 使模型能更准确地生成相似的结果
```

### V3_STRUCTURED（结构化输出）

```python
【特点】
- 明确定义JSON Schema
- 强制字段类型检查
- 详细的判断标准

【提示词框架】
1. 严格的Schema定义
2. 必选值列表
3. 详细的分析标准
4. 格式要求强调

【添加的字段】
{
    "fraud_indicators": {
        "urgency": "低/中/高",
        "financial_request": "是/否",
        "identity_claim": "是/否",
        "trust_exploitation": "是/否"
    }
}

【优点】
✓ 输出格式完全可控
✓ 易于程序解析
✓ 包含额外的判断维度

【缺点】
✗ 响应可能不如预期一致
✗ 有时模型难以遵循严格的Schema
✗ 调试困难

【何时使用】
- 需要严格的输出格式
- 自动化系统需要特定字段
```

### V4_CONTEXTUAL（上下文感知 - 最优）

```python
【特点】
- 包含诈骗特征库
- 考虑上下文信息
- 多维度分析

【提示词框架】
1. 【关键诈骗类型特征库】
   - 冒充公检法：特征列表
   - 投资理财：特征列表
   - ... 等7种

2. 【上下文因素】
   - 相似案例的通用特征
   - 文本中的紧迫性语言
   - 涉及的金额
   - 对方的身份声称

3. 详细分析框架

【输出字段】
{
    "analysis": {
        "detected_patterns": ["模式1", "模式2"],
        "suspicious_phrases": ["短语1", "短语2"],
        "urgency_indicators": ["指标1", "指标2"],
        "trust_exploitations": ["利用1", "利用2"]
    }
}

【优点】
✓ 准确性最高（90%+）
✓ 分析最深入
✓ 考虑了更多因素
✓ 符合专家思维

【缺点】
✗ 响应时间长（2-3秒）
✗ API调用成本更高
✗ 需要更多上下文信息

【何时使用】
- 高价值用户
- 高风险检测
- 需要最高准确性
- 可以接受较慢的响应时间
```

## 少样本学习

### 原理

```
示例的作用：
1. 提供行为模式
2. 演示推理过程
3. 规范输出格式
4. 建立上下文理解

大模型的学习过程：
输入 → 看到示例 → 理解任务风格 → 按风格输出 → 一致的结果
```

### 设计高质量示例的要点

#### ✓ 示例1：典型的高风险情况

```json
{
    "pattern": "冒充公检法 + 要求转账",
    "risk_level": "高",
    "confidence": "很高（0.98）"
}
```

**特征**：
- 冒充身份（公安局）
- 虚构罪名（涉嫌洗钱）
- 紧迫语言（立即）
- 资金要求（转账）

#### ✓ 示例2：典型的低风险情况

```json
{
    "pattern": "日常闲聊",
    "risk_level": "低",
    "confidence": "很高（0.95）"
}
```

**特征**：
- 日常交流内容
- 没有经济交易
- 没有紧迫性
- 完全正常

### 为什么要同时提供高风险和低风险示例

1. **对比学习**：模型通过对比理解"什么是诈骗"和"什么不是诈骗"
2. **边界定义**：帮助模型划分风险等级的临界点
3. **减少误判**：低风险示例防止模型过度判断
4. **一致性**：示例越丰富，输出越一致

### 示例与实际的映射

```
示例设计应该考虑：
✓ 覆盖主要诈骗类型
✓ 绘制了边界情况
✓ 大致代表实际分布
✓ 难度适中（不要太简单也不要太复杂）
```

## 调优技巧

### 1. 调整系统角色

**当前设置**：
```
"你是一位资深的反诈专家，拥有20年的反诈经验"
```

**调优选项**：
```
# 更严格
"你是一位极其谨慎的反诈专家，误杀不避，宁可高判也不低判"

# 更平衡
"你是一位经验丰富的反诈专家，需要平衡准确性和用户体验"

# 针对特定角色
"你是一位儿童安全保护专员，对未成年人的安全判断要特别严格"
```

### 2. 调整分析标准

**当前**：
```
1. 识别常见的诈骗模式
2. 识别紧急警告词
3. 分析文本的紧迫性
4. 评估信息的可信度
```

**更细粒度的**：
```
1. 有无身份冒充？(权重: 40%)
2. 有无资金要求？(权重: 35%)
3. 有无紧迫性语言？(权重: 15%)
4. 有无敏感信息请求？(权重: 10%)
```

### 3. 调整温度参数

```python
temperature=0.1  # 更确定的输出（推荐用于诈骗检测）
temperature=0.3  # 平衡的输出（推荐）
temperature=0.7  # 更多样化的输出

# 设置
response = openai.ChatCompletion.create(
    temperature=0.3,  # ← 关键参数
    ...
)
```

### 4. 调整输出格式

```python
# 添加格式约束
"请返回严格的JSON格式，确保：
 1. 所有字符串必须用双引号
 2. 所有数字必须在0-100范围内
 3. risk_level 必须是 '低'、'中' 或 '高'"
```

### 5. 逐步细化（Chain of Thought）

```python
# 添加思考过程
"""
分析过程：
1. 首先，识别是否涉及身份冒充 → [分析结果]
2. 其次，是否包含资金要求 → [分析结果]
3. 再次，有无紧迫性语言 → [分析结果]
4. 最后，综合以上因素得出风险等级 → [最终结果]
"""
```

## 常见问题解决

### 问题1：输出格式不一致

**症状**：有时返回的JSON格式不同，某些字段缺失

**解决方案**：
```python
# 添加更严格的格式要求
"必须返回以下结构的有效JSON：
{
    \"fraud_type\": \"string\",
    \"risk_level\": \"string (low/medium/high)\",
    \"risk_score\": \"number (0-100)\",
    \"confidence\": \"number (0-1)\",
    \"reasons\": \"array of strings\",
    \"warning_keywords\": \"array of strings\",
    \"advice\": \"string\"
}

所有字段都必须存在。如果不知道某个值，也要返回合理的默认值。"
```

### 问题2：准确性不足（误判或漏判）

**症状**：错误地将正常消息分类为诈骗，或错过真实诈骗

**解决方案**：
```python
# 添加更多和更好的示例
# 提供行业特定的案例
# 增加分析维度
# 调整权重
```

### 问题3：置信度偏高

**症状**：模型总是返回很高的置信度，即使答案可能不准确

**解决方案**：
```python
"置信度应该反映你的真实确定程度：
- 0.95-1.0: 非常确定的判断（明显的诈骗线索）
- 0.75-0.95: 相当确定（多个风险指标）
- 0.50-0.75: 中等确定（一些风险指标）
- 0.30-0.50: 不确定（模糊或矛盾的信号）
- <0.30: 非常不确定（不足以做出判断）"
```

### 问题4：响应时间过长

**症状**：V4_CONTEXTUAL 版本太慢

**解决方案**：
```python
# 方案1: 使用更快的模型
model_name = "gpt-3.5-turbo"  # 比 gpt-4 快5倍

# 方案2: 简化提示词
# 从V4降级到V2

# 方案3: 异步处理
# 使用后台任务队列

# 方案4: 并发调用
# 批量分析多个文本
```

### 问题5：成本过高

**症状**：API 调用费用超出预算

**解决方案**：
```python
# 1. 缓存结果
cache.set(text_hash, result, ttl=86400)  # 24小时缓存

# 2. 使用本地规则预筛选
if is_obviously_safe(text):
    return quick_result()  # 跳过LLM调用

# 3. 采样分析
if random.random() > 0.1:  # 只分析10%
    return cached_result()

# 4. 使用更便宜的模型
model_name = "gpt-3.5-turbo"  # 便宜75%

# 5. 批量处理
results = batch_analyze(texts)  # 批量API调用
```

### 评估指标

```python
# 准确率
accuracy = correct_predictions / total_predictions

# 精确率（高风险中真正的诈骗比例）
precision = true_positives / (true_positives + false_positives)

# 召回率（检测出的诈骗占全部诈骗的比例）
recall = true_positives / (true_positives + false_negatives)

# F1分数（平衡精确率和召回率）
f1 = 2 * (precision * recall) / (precision + recall)
```

## 最佳实践总结

| 方面 | 建议 |
|-----|------|
| **选择版本** | 使用 V2_EXAMPLES（少样本学习） |
| **性能优化** | 使用缓存、异步和批处理 |
| **成本控制** | 使用本地规则预筛选 |
| **准确性** | 为高危用户使用 V4_CONTEXTUAL |
| **测试** | 定期在真实数据上测试 |
| **监控** | 监控准确性和成本 |
| **迭代** | 根据用户反馈不断优化 |
